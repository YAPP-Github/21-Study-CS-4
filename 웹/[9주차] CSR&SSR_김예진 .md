# CSR & SSR

# CSR (Client Side Rendering)

- 렌더링이 클라이언트 쪽에서 일어난다.
- 서버는 요청을 받으면 클라이언트에 HTML과 JS를 보내준다.
- 클라이언트는 이 파일을 받아 렌더링을 시작한다.

## CSR 동작과정

![Untitled](https://user-images.githubusercontent.com/68051794/217755222-87508ab7-c591-4b28-a0fe-0df636249637.png)

1. User가 웹사이트 요청을 보냄.
2. CDN이 HTML 파일과 JS로 접근할 수 있는 링크를 클라이언트로 보낸다.
3. 클라이언트는 HTML과 JS를 다운로드 받는다.

   → 이때 유저는 아무것도 볼 수 없다.

4. 브라우저가 자바스크립트를 다운로드 받는다.
5. 다운로드가 완료된 JS가 실행된다. 데이터를 위한 API가 호출된다.

   → 이때 유저는 placeholder들을 볼 수 있다.

6. 서버가 API로부터의 요청에 응답한다.
7. API로부터 받아온 data를 placeholder 자리에 넣어준다.

   → 해당 페이지에서 상호작용이 가능해진다.

![Untitled 1](https://user-images.githubusercontent.com/68051794/217755267-a3d5bd8a-1944-405d-b90a-8fe805311055.png)

즉, 서버에서 아무런 처리 없이 클라이언트로 보내주기 때문에 자바스크립트가 모두 다운로드 되고 실행이 끝나기 전까지는 사용자가 볼 수 있는 것이 없다.

# SSR (Server Side Rendering)

- 서버에서 렌더링 준비를 끝마친 상태로 클라이언트에 전달하는 방식.

### SSR 실행단계

![Untitled 2](https://user-images.githubusercontent.com/68051794/217755321-492d0341-0795-4b11-a96b-a80bb4bf98fa.png)

1. User가 Website 요청을 보냄.
2. Server는 렌더링 가능한 HTML 파일을 만든다.
3. 클라이언트에 전달 되는 순간, 이미 렌더링 준비가 되어있기 때문에 HTML은 즉시 렌더링 된다.

   → 그러나 사이트는 조작(상호작용) 불가능한 상태이다.

4. 클라이언트가 자바스크립트를 다운받는다.
5. 유저는 내용을 볼 수 있으나 사이트 조작은 할 수 없다. 이 때 사용자의 조작을 기억하고 있는다.
6. 브라우저가 JS 프레임워크를 실행한다.
7. JS가 성공적으로 컴파일 된 후 기억하고 있던 사용자 조작이 실행된다.

   → 웹 페이지에서 상호작용이 가능해진다.

![Untitled 3](https://user-images.githubusercontent.com/68051794/217755373-8e94df2c-a10f-40f6-b73b-4f2485c6f77a.png)

즉, 서버에서 이미 ‘렌더 가능한’ 상태로 클라이언트에 전달되기 때문에, JS가 다운로드 되는 동안 사용자는 무언가를 보고 있을 수 있다.

# CSR vs SSR

### 1. **웹 페이지를 로딩하는 시간**

- 첫 페이지 로딩 시간
  - CSR은 HTML, CSS, 모든 JS를 한번에 불러옴.
  - SSR은 필요한 부분의 HTML과 스크립트만 불러오게 된다.
  - 따라서 평균적으로 SSR이 더 빠르다.
- 나머지 로딩 시간
  - 첫 페이지 로딩 후 사이트의 다른 곳으로 이동하려고 하면, CSR은 이미 나머지 부분을 구성하는 코드를 받아왔기 때문에 빠르다.
  - 반면 SSR은 첫 페이지를 로딩한 과정을 다시 실행하며, 따라서 비교적 느리다.

### 2. **SEO 대응**

- 검색 엔진은 자동화된 로봇인 ‘크롤러’로 웹사이트들을 읽는데, CSR은 JS를 실행시켜 동적으로 컨텐츠가 생성되기 때문에 JS가 실행 되어야 metadata가 바뀐다.
- SSR은 애초에 서버 사이드에서 컴파일되어 클라이언트로 넘어오기 때문에 크롤러에 대응하기 용이하다.

### 3. **서버 자원 사용**

- SSR은 매번 서버에 요청을 해야하기 때문에 서버의 자원을 더 많이 사용한다.
- 반면 CSR은 클라이언트에게 부담이 더 가기 때문에 서버의 부하가 적다.
